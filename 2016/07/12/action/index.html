Spring in action

程序开发
  • Spring


快速的学习Spring in action ！！！

Spring

Spring的出现就是为使Java EE应用开发更加简单。整个Spring主要是围绕下面4点来简化开发的的：

  • 基于POJO的轻量级和最小侵入性编程
  • 通过依赖注入和面向接口实现松耦合
  • 通过切面和惯例进行声明式编程
  • 通过模版减少样板式代码

基本上，Spring做的所有的事都可以回溯到这上面来。接下来我会举例来为大家说明。

POJO

因为Spring鼓励使用POJO，所以它是一种轻量级和最小侵入性编程。具体原因
详细了解POJO后，你就会明白，这里建议理清POJO和JavaBean的区别。
POJO（Plain Old Java Object）就像他的名字一样，他就是一个最普通的java对象。那么为什么说基于Spring的应用是轻量级和最小侵入性的呢？

  • Spring不会强迫你实现Spring规范的接口或继承Spring规范的类，从而使你根本看不出来声明的类到底有没有使用Spring。
  • Spring可以通过DI来装载POJO

代码中的DI

DI就是依赖注入，AOP就是面向接口的编程。在这里不做过多介绍，下面将通过一个例子来具体说明。

public class DamselRescuingKnight
{

    private RescueDamselQuest quest;

    public DamselRescuingKnight() 
    {
        this.quest = new RescueDamselQuest();
    }

    public void embarkOnQuest()
    {
        quest.embark();
    }
}


public class RescueDamselQuest
{

    public void embark()
    {
        System.out.println("embark on the quest of Rescuing Damsel");
    }
}

看到你各类你会感觉很不符合规范，那么哪里不符合规范呢？

问题一：

这里有两个类DamselRescuingKnight和RescueDamselQuest，看到这些我们会想：如果我要使用DamselRescuingKnight，我就得想初始化一个RescueDamselQuest对象，并赋值给DamselRescuingKnight中的quest。这两个类联系也太紧密了，这就是我们所说的强藕连（tightly coupling）。这样的类是根本没法测试。一般情况下，单元测试基本不涉及类之间方法的操作调用。而这里若对embarkOnQuest方法进行测试，就必须要调用embark方法。那么怎么解决这个问题呢？ 很简单，提取一个接口就可以。

这里我们看到没有创建自己的quest，它是通过构造函数来给quest赋值的,其实这也是DI的一种：通过构造函数来注入。另外这里的quest是接口，你可以声明任何类来继承该接口来给这里的quest赋值，从而也就实现了解藕（loose coupling）。那么我们这里怎么来写单元测试呢？只需mock一个继承Quest接口就可以了。

AOP

什么是AOP呢？AOP就是Aspect Oriented Programming，貌似是废话。我们先不要去管AOP的定义，不是有句话是说：讲道理不如举例子。这里我们就先讲一个例子。

一个模块除了要执行他的核心代码之外，可能还要执行一些辅助性的代码：比如日志打印，安全性校验，检查事物状态等。

如果想在系统多处执行这个核心功能，我们就不得不也在多个位置加入相应的辅助性代码。这一方面造成了许多重复代码的出现，一方面更不利于我们维护，修改该模块代码。（这里我们就不能把这些模块原子化吗？在有的时候辅助性的代码是和类相关的，如日志代码。在这种情况下，这些模块是不可能被原子化）

为了解决这样的问题，AOP就应运而生了。AOP就有这一个作用，那就是使核心代码与其他代码分隔开。我们为上面的knight配备一个Minstrel，使得每次knight在执行任务的时候，minstrel都会为他咏唱一下。

总结

从上面介绍POJO，DI，AOP，使用接口我们可以看出，Spring和配合Spring来使用的一些技术，都是为了使我们的程序在代码层面显得更简练，在逻辑层面更清晰，在系统层面更加的模块化。这些基本上就是以Spring框架来搭建的系统的最可取的地方。
